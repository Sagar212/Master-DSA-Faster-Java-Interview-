Architectural Design: High-Traffic Read-Heavy Catalog

Scenario: You have a high-traffic e-commerce catalog microservice using Spring Boot 3.x and Hibernate 6.x. Data changes infrequently but reads are extremely high. Initially, Redis with @Cacheable was used, but under massive spikes, Redis became a network bottleneck, and "cache stampedes" crushed the database when TTL expired.

How do you optimize this using modern paradigms (as of 2026), tackle the stampede, and handle cache invalidation effectively without DB polling?

  1. Multi-Level Caching (L1 + L2): Relying solely on Redis (L2) introduces network latency. We implement an L1 in-memory cache (like Caffeine) for ultra-fast, zero-network overhead. A request first checks Caffeine, then Redis, and finally the DB.
  2. Cache Stampede Prevention: Instead of simple TTL expiration (expireAfterWrite) which causes all concurrent requests to miss at once, we use refreshAfterWrite in Caffeine coupled with asynchronous cache loading. Only one thread fetches the updated value from the DB while others are served the stale value, fully shielding the database. In Spring, we can also use @Cacheable(sync = true) to lock the key on cache misses.
  3. Event-Driven Cache Invalidation (CDC): TTLs lead to stale data. Polling the DB is inefficient. We use Change Data Capture (CDC) like Debezium. When a product updates, Debezium reads the database transaction log (WAL) and publishes an event to Kafka. Our microservice consumes this event and evicts the specific key from both L2 (Redis) and L1 (Caffeine). Since L1 is local to each instance, we broadcast the invalidation via Redis Pub/Sub to all pods.
  4. Spring Boot 3 & Java 21+ Virtual Threads: We enable Project Loom's Virtual Threads (spring.threads.virtual.enabled=true). For any cache misses that require a blocking DB call, Virtual Threads prevent OS thread starvation, allowing the service to handle tens of thousands of concurrent connections with a minimal memory footprint.
  5. Hibernate 6 Optimizations: We utilize Hibernate 6's semantic query engine and array batching capabilities. Since reads are heavy, we map @NaturalId and leverage Hibernate's 2nd-Level Cache synced with our L2 strategy if entity caching is preferred over DTO caching.
Tricky Part / Interview Trap: Interviewers will often poke holes in multi-level caches by asking about cache coherence across instances. If Pod A updates the DB and clears its local cache, Pod B still has the old data in its local Caffeine cache. Mentioning the Pub/Sub broadcasting pattern to notify all instances to evict their L1 cache is the hallmark of a Senior/Architect answer.
