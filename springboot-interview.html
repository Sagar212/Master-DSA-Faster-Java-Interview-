Spring Boot: Core Concepts & Internals

Detailed, must-know questions for cracking mid to senior level Spring Boot interviews.

1. How exactly does @SpringBootApplication work internally?

The @SpringBootApplication annotation is actually a combination of three key annotations:

  • @Configuration: Designates the class as a source of bean definitions for the application context.
  • @ComponentScan: Tells Spring to look for other components, configurations, and services in the current package and its sub-packages, automatically registering them as beans.
  • @EnableAutoConfiguration: This is the true magic of Spring Boot. It tells Spring to start adding beans based on classpath settings, other beans, and various property settings. For instance, if spring-webmvc is on the classpath, this annotation flags the application as a web application and activates key behaviors like setting up a DispatcherServlet.
Tricky Part: Interviewers will ask "What happens if you move a @Service class to a package OUTSIDE the folder where your @SpringBootApplication class is?" Answer: Spring won't find it! You must explicitly add @ComponentScan(basePackages = {"com.other.package"}).
2. Bean Scopes and Thread Safety: Is a Spring @Service thread-safe?

By default, all Spring Beans are Singleton scope. This means only one instance of the @Service or @Controller is created per Application Context. It is shared across all incoming HTTP requests (which run on separate threads).

Therefore, Spring Beans are NOT inherently thread-safe. If you store state (like a class-level mutable variable private int counter = 0;) in a Service or Controller, concurrent requests will cause race conditions.

Tricky Part: To make them thread-safe, you must keep them stateless. Never use class-level variables to hold request-specific data. If you absolutely need state, you must change the bean scope to @Scope("request") or use ThreadLocal.
3. Explain the request lifecycle in Spring Boot (DispatcherServlet).

Spring MVC uses the Front Controller Pattern. The lifecycle is:

  1. An HTTP request hits the DispatcherServlet (the front controller).
  2. The DispatcherServlet consults the HandlerMapping to find the appropriate @Controller/@RestController method that matches the URL.
  3. It then routes the request to the Controller via a HandlerAdapter.
  4. The Controller processes the business logic (usually calling a Service) and returns data.
  5. If it's a @RestController, the data is passed through HttpMessageConverters (like Jackson) to convert the Java object into JSON directly into the HTTP response body.
Tricky Part: Knowing the difference between @Controller and @RestController. @RestController is simply a meta-annotation that combines @Controller + @ResponseBody. Without @ResponseBody, Spring assumes you are returning the name of a view template (like Thymeleaf/JSP) instead of raw JSON!
4. What is Spring Boot Actuator and why is it essential for microservices?

Spring Boot Actuator brings production-ready features to our application out of the box without us having to write code for them. It exposes operational information about the running application via HTTP endpoints (e.g., /actuator/*).

  • /health: Shows application health information. Kubernetes uses this for Liveness and Readiness probes to automatically restart failed pods or stop sending traffic to them.
  • /metrics: Exposes metrics like memory usage, garbage collection, and HTTP request timings. Can be scraped by Prometheus.
  • /env: Exposes properties from Spring's ConfigurableEnvironment.
Tricky Part: Security! Actuator endpoints expose sensitive data. In Spring Boot 2.x/3.x, only /health and /info are exposed over HTTP by default. You must explicitly enable others via management.endpoints.web.exposure.include=* and secure them using Spring Security.
5. Dependency Injection: Constructor Injection vs @Autowired Field Injection

Spring allows injecting dependencies via Fields (using @Autowired on a private field), Setter methods, or Constructors.

Constructor Injection is the highly recommended approach:

  • Immutability: Dependencies can be declared as final, ensuring they are not changed after initialization.
  • Testing: You can easily instantiate the class in unit tests using the constructor without needing Spring or reflection (Mockito's @InjectMocks).
  • Startup Failures: Missing dependencies throw an error immediately on application startup, rather than a NullPointerException randomly at runtime.
Tricky Part: Since Spring 4.3, if a class only has one constructor, you don't even need to write the @Autowired annotation. Spring will automatically use that constructor for injection.
