Accolite Digital L1 Interview Guide

Java Backend Developer — Full Production-Grade Q&A

Section 1: Java Core, Strings, and Thread Safety

Q1 String Memory & Immutability
Strings are immutable to ensure security (database URLs/passwords can't be altered), thread safety (can be shared safely across threads), and caching performance (String pool relies on immutability to save memory).
Q2 String Constant Pool Internals
The String Pool is a special area in the JVM Heap. When creating a String via String s = "Hello", the JVM checks the pool first. If it exists, it returns the reference, otherwise it creates it. Using new String("Hello") forces creation of a new object in the general heap, bypassing the pool.
Q3 Exception Handling (try-with-resources)
Introduced in Java 7, try-with-resources automatically closes any resources that implement the AutoCloseable interface. This prevents memory leaks and removes the need for messy finally blocks containing close() calls.
Q4 Volatile vs Synchronized
volatile guarantees visibility of changes to variables across threads by forcing reads/writes from main memory instead of CPU caches. It does NOT guarantee atomicity. synchronized guarantees both visibility and atomicity by acquiring a monitor lock, allowing only one thread to execute the block at a time.
Q5 ReentrantLock vs Synchronized
ReentrantLock offers advanced features over synchronized: fairness policies (longest waiting thread gets the lock), tryLock() (attempt lock without blocking forever), and interruptible locks. synchronized is simpler and handled implicitly by the JVM.
Q6 Object-Level vs Class-Level Locking
Object-level lock (synchronized(this)) restricts concurrent access to the instance methods of a specific object. Class-level lock (synchronized(MyClass.class) or static synchronized) restricts concurrent access across ALL instances of the class.
Q7 ConcurrentHashMap Treeification
In Java 8+, when a bucket in a HashMap/ConcurrentHashMap exceeds a certain threshold (usually 8 nodes) and total capacity > 64, the linked list is converted into a Red-Black Tree (Treeification). This improves worst-case lookup performance from O(n) to O(log n).
Q8 Fail-Fast vs Fail-Safe Iterators
Fail-Fast iterators (e.g., ArrayList, HashMap) throw ConcurrentModificationException if the collection is structurally modified during iteration. Fail-Safe iterators (e.g., ConcurrentHashMap, CopyOnWriteArrayList) operate on a clone of the collection or use lock-free mechanisms, never throwing the exception.
Q9 HashMap vs ConcurrentHashMap Internal Differences
HashMap is not thread-safe. ConcurrentHashMap is thread-safe. In Java 8, CHM abandoned segment locking in favor of Node-level locking (using synchronized blocks on the head node of a bucket) and CAS operations, allowing vastly higher concurrency.
Q10 Callable vs Runnable
Runnable has a run() method that returns void and cannot throw checked exceptions. Callable has a call() method that returns a result (via a Future) and can throw checked exceptions.

Section 2: Object Orientation, Cloning & Design Principles

Q11 Shallow vs Deep Copy
A Shallow Copy copies primitive fields but copies only the memory addresses of nested objects (they share the same nested objects). A Deep Copy recursively creates exact copies of all nested objects so the two instances are completely independent.
Q12 Object.clone() internally
The default Object.clone() provides a shallow copy. To use it, a class must implement the marker interface Cloneable, otherwise it throws CloneNotSupportedException.
Q13 Abstract Class vs Interface (Java 8+)
Interfaces can now have default and static methods. However, abstract classes can still hold instance state (fields/constructors) and have varied access modifiers (protected/private methods). A class can implement multiple interfaces but extend only one abstract class.
Q14 Method Hiding vs Overriding
Method Overriding applies to instance methods and is resolved at runtime (Dynamic Polymorphism). Method Hiding applies to static methods; the child class hides the parent's static method, but resolution happens at compile-time based on the reference type.
Q15 SOLID Principles in real projects
Single Responsibility (Classes do one thing), Open/Closed (Extensible via interfaces, closed for modification), Liskov Substitution (Derived classes can substitute parents without breaking code), Interface Segregation (Small, specific interfaces), Dependency Inversion (Depend on abstractions, not concretions).

Section 3: Database, Hibernate, and Querying

Q16 FetchType.LAZY vs EAGER
EAGER fetches related entities immediately along with the parent. LAZY delays fetching until the related entity is explicitly accessed via a getter, improving initial load performance but risking LazyInitializationException outside a transaction.
Q17 L1 vs L2 Cache in Hibernate
L1 Cache is session-scoped, enabled by default, and cleared when the Session closes. L2 Cache is SessionFactory-scoped, shared across all sessions, and requires an external provider (like EhCache or Redis) to be configured.
Q18 N+1 Select Problem
Occurs when querying a collection of N parents triggers 1 query to load the parents, and then N additional queries to load their LAZY children. Solved using JOIN FETCH in JPQL or Entity Graphs to fetch everything in a single query.
Q19 Save vs SaveOrUpdate vs Persist
persist() makes a transient instance persistent and void return type. save() does the same but returns the generated identifier. saveOrUpdate() executes an INSERT if the entity is transient, or an UPDATE if it is detached.
Q20 Optimistic vs Pessimistic Locking
Optimistic locking (using @Version) assumes conflicts are rare; it checks version numbers on update and throws an exception on mismatch. Pessimistic locking acquires actual database row-level locks (SELECT FOR UPDATE) preventing other transactions from reading/writing until committed.
Q21 Indexing (B-Tree vs Hash)
B-Tree indexes keep data sorted, allowing efficient range queries (>, <, BETWEEN). Hash indexes use a hash function to map keys directly to buckets, making exact-match lookups (=, IN) O(1) but cannot support range queries.
Q22 Transaction Isolation Levels
Read Uncommitted (Lowest, dirty reads), Read Committed (Prevents dirty reads), Repeatable Read (Prevents non-repeatable reads), Serializable (Highest, locks entire ranges, prevents phantom reads but kills concurrency).
Q23 Dirty Read, Phantom Read, Non-Repeatable Read
Dirty Read: Reading uncommitted changes. Non-Repeatable Read: Same query yields different data because another transaction updated it. Phantom Read: Same query yields different number of rows because another transaction inserted/deleted records.

Section 4: Spring Framework, Spring Boot & AOP

Q24 Spring Boot Autoconfiguration
Powered by @EnableAutoConfiguration, Spring Boot scans the classpath and automatically configures beans based on detected dependencies. For example, if spring-webmvc is found, it automatically sets up a DispatcherServlet and embedded Tomcat.
Q25 @Controller vs @RestController
@Controller maps to View Resolvers (returning HTML pages). @RestController is a convenience annotation combining @Controller and @ResponseBody, meaning methods directly serialize objects into JSON/XML via HttpMessageConverters.
Q26 Bean Scopes & Default Scope
The default is Singleton (one instance per Spring IoC container). Prototype creates a new instance every time it's injected. Request, Session, and Application scopes are specific to web-aware application contexts.
Q27 @Transactional propagation and rollback rules
By default, Spring rolls back transactions ONLY for unchecked exceptions (RuntimeExceptions). To rollback for checked exceptions, use @Transactional(rollbackFor = Exception.class). Propagation REQUIRED joins an existing transaction or creates a new one. REQUIRES_NEW suspends the current transaction and forces a new one.
Q28 AOP (Aspect Oriented Programming)
AOP modularizes cross-cutting concerns (logging, security, transactions) separately from business logic. Key concepts: Aspect (the module), Pointcut (expression matching where advice should run), Advice (the action taken: Before, After, Around), JoinPoint (the actual execution point in the code).

Section 5: Microservices & Distributed Systems

Q29 API Gateway pattern
A single entry point for all client requests. It handles cross-cutting concerns like authentication, SSL termination, rate limiting, and request routing/aggregation to backend microservices, preventing direct coupling between clients and internal services.
Q30 Circuit Breaker (Resilience4j)
Prevents cascading failures by "tripping" when a downstream service fails repeatedly. State machine: CLOSED (requests flow normally), OPEN (fails fast without calling the failing service), HALF-OPEN (allows a few test requests to see if the service recovered).
Q31 Saga Pattern (Choreography vs Orchestration)
Manages distributed transactions. Choreography: Services independently listen and emit events (decentralized, hard to track). Orchestration: A central controller service dictates the sequence of local transactions and triggers compensating transactions on failure.
Q32 Event-Driven Architecture with Kafka
Services communicate asynchronously via a broker (Kafka). Producers publish events to Topics; Consumers subscribe to Topics. This decouples services, handles traffic spikes via buffering, and allows replayability of events.

Section 6: Caching & Performance

Q33 Redis data structures
Redis supports Strings (simple key-value caching), Hashes (storing objects), Lists (queues), Sets (unique elements, fast intersection), and Sorted Sets (leaderboards, ranking by score).
Q34 Cache Aside vs Write-Through
Cache Aside: Application reads from cache; on miss, reads from DB and updates cache. Write-Through: Application writes to cache, and the cache synchronously writes to the DB, ensuring consistency at the cost of write latency.
Q35 Eviction Policies (LRU/LFU)
When cache is full, LRU (Least Recently Used) evicts the oldest accessed items. LFU (Least Frequently Used) tracks access counts and evicts the least popular items. Redis often uses volatile-lru.

Section 7: System Design Scenario

Q36 Design a Rate Limiter
Use a distributed caching system like Redis to store limits per User/IP. Algorithms: Token Bucket (add tokens at constant rate, allow request if token exists), Leaky Bucket (queue requests, process at constant rate), or Sliding Window Log (store timestamps in Redis Sorted Sets, drop old timestamps, count size). Sliding window log provides accurate throttling.
