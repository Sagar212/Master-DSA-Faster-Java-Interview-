Accolite L1 Interview Guide

Exhaustive, production-grade answers to Core Java, Spring Boot, and System Design.

Section 1: Core Java
1. Why is String immutable in Java?
Immutability ensures that once a String object is instantiated, its state cannot be modified. Java enforces this for:

  • String Constant Pool (SCP) Optimization: Memory efficiency is achieved through string sharing.
  • Security Constraints: Strings are relied upon for file paths, DB connections, etc. Immutability prevents Time-of-Check to Time-of-Use flaws.
  • Thread Safety: Inherently thread-safe because state cannot be modified, eliminating synchronization overhead.
  • HashCode Caching: The hash code is cached at creation, making Strings extremely high-performing as keys in HashMaps.
2. Internal Working of HashMap in Java 8
A HashMap uses an internal array of nodes (buckets) to store key-value entries. In Java 8, Treeification was introduced:

  • Data Structures: Elements are tracked as instance nodes. Extreme hash collisions transition the single-linked list into a balanced Red-Black Tree.
  • Treeification Mechanics: If bucket nodes exceed TREEIFY_THRESHOLD = 8 and table capacity >= 64, it transforms into a Red-Black tree, reducing search time from O(n) to O(log n). Untreeifies at 6.
  • Hash Distribution: Uses bitwise XOR (h ^ (h >>> 16)) to mix higher-order bits downward, eliminating bucket distribution clustering.
3. Difference between HashMap and ConcurrentHashMap
  • Thread Safety: HashMap is not thread-safe. CHM is fully thread-safe for high concurrent modifications.
  • Locking Strategy: HashMap has no synchronization. CHM uses fine-grained bucket-level locking (CAS for empty buckets, synchronized blocks on collision heads).
  • Null Support: HashMap permits one null key and multiple null values. CHM prohibits both null keys and values (throws NullPointerException).
  • Iterator Behavior: HashMap is Fail-Fast (throws ConcurrentModificationException). CHM is Fail-Safe (weakly consistent).
4. Explain equals() and hashCode() Contract
1. If two objects are equal via equals(), executing hashCode() on both must return the exact same integer value.
2. If two objects yield matching integers via hashCode(), they are not required to be equal (this is a hash collision).

Consequence: If equals() is customized but hashCode() is ignored, HashMaps will locate different buckets during retrieval, causing duplicate keys or null returns.
5. Can we override static methods?
No. Method overriding is dependent on dynamic execution binding at runtime based on the object instance. Static methods are bound at compile-time (static binding).

If a child class declares an identical static signature, it is termed Method Hiding. The executing method is determined strictly by the reference type declared on the caller variable.
6. Difference between Abstract Class and Interface
  • Inheritance: Abstract classes bind by single inheritance. Interfaces allow multiple implementation simultaneously.
  • State: Abstract classes can declare instance variables with varying scope. Interfaces only declare public static final constants.
  • Constructors: Abstract classes permit constructors. Interfaces cannot have constructors.
  • Methods: Abstract classes have abstract, concrete, protected, or private routines. Interfaces have abstract, default, static, and private helpers (Java 8+).
7. What is the purpose of volatile?
  • Memory Visibility: Instructs JVM to bypass local CPU caches and read/write directly from main memory.
  • Instruction Reordering Inhibition: Places memory fences around reads/writes, establishing a strict 'happens-before' contract.
  • Non-Atomicity Caveat: Does not make compound operations (like count++) atomic. Still requires locks or Atomic wrapper classes.
Section 2: Java 8 Stream API
11. Difference between map() and flatMap()
  • map(): 1-to-1 mapping transformation. Outputs a single processed value of type R.
  • flatMap(): 1-to-many structural transformation combined with flattening. Useful for unwrapping nested collection matrices (e.g., Stream<List<String>> to Stream<String>).
15. Difference between Optional.orElse() and orElseGet()
  • orElse(T other): Evaluates argument eagerly. Fallback instantiation runs even if Optional is non-empty, causing overhead.
  • orElseGet(Supplier): Implements lazy evaluation. Lambda is only evaluated if the Optional is explicitly empty.
Section 3: System Design & Production
20. LRU Cache Design
An LRU cache requires O(1) operations. Achieved using a HashMap paired with a custom Doubly Linked List. The map provides O(1) lookups, while the linked list allows O(1) node detachment and moving to the head (most recently used).
24. What happens internally when Spring Boot starts?
1. Initializer Setup: Identifies profiles and configurations via spring.factories.
2. Environment Preparation: Binds active properties and system variables.
3. ApplicationContext Creation: Instantiates the core container.
4. Component Scanning: Parses stereotype markers to construct BeanDefinitions.
5. Embedded Container: Bootstraps Tomcat or Jetty.
6. Runner Invocation: Executes CommandLineRunner beans.
28. How does @Transactional work internally?
Spring manages transactions using AOP Proxies. When a caller executes a method, it hits the auto-generated proxy first. The proxy opens a DB transaction via PlatformTransactionManager, invokes the target method in a try-catch, and commits on success (or rolls back on RuntimeException).
36. Explain one complex production issue you resolved
Issue: High-traffic Promotional Event caused HikariPool connection timeouts and JVM OutOfMemoryErrors.

Root Cause: Heap dump analysis via Eclipse MAT revealed a reporting feature streaming huge datasets inside an active @Transactional block. The DB stream remained open, holding the connection indefinitely, starving the pool and exhausting heap space.

Resolution: Refactored retrieval using Spring Data Pageable for chunking, moved processing outside the @Transactional block, wrapped access in try-with-resources, and enabled HikariCP leak-detection.
