Docker & Kubernetes: Must Know & Tricky

Containerization and orchestration questions covering optimizations, ephemeral architecture, and security.

1. What is the difference between COPY and ADD in a Dockerfile?

Both instructions copy files from the host to the container. However, ADD has two extra features: it can automatically extract local tar files into the image, and it can download files from a remote URL.

Tricky Part: Best practice dictates that you should always use COPY unless you explicitly need the auto-extraction feature of ADD. Using ADD for remote URLs is discouraged because it adds an extra layer without letting you delete the archive to save space.
2. How do CMD and ENTRYPOINT differ, and how do they work together?

ENTRYPOINT specifies the executable that will always run when the container starts. CMD provides default arguments to that executable, or specifies a default command if no entrypoint is set.

Tricky Part: If you use the "shell form" (e.g., ENTRYPOINT node app.js instead of the array form ENTRYPOINT ["node", "app.js"]), Docker wraps it in /bin/sh -c. This means your app won't run as PID 1 and won't receive OS signals (like SIGTERM), causing graceful shutdown to fail.
3. What are the best strategies to reduce Docker image size?

Use smaller base images like Alpine linux (alpine). Combine multiple RUN commands into a single layer to reduce intermediate layers. Use Multi-stage builds to compile code in one stage and only copy the final binaries into a minimal production stage.

Tricky Part: Removing temporary files in a subsequent RUN layer does NOT reduce the image size! The previous layer still contains the files. You must clean up the cache and temp files in the same RUN command where they were created.
4. In Kubernetes, what happens to the underlying Pods when a Deployment is updated?

Kubernetes performs a Rolling Update. It creates a new ReplicaSet, spins up new Pods based on the new template, and gradually scales down the old ReplicaSet. Pods are immutable; they are never "restarted" or updated in place.

Tricky Part: Pods are fundamentally ephemeral. Never store local state in a Pod expecting it to persist. If a node crashes, the Pod dies permanently, and a brand new Pod is scheduled elsewhere.
5. What is the difference between ClusterIP, NodePort, and LoadBalancer services?

ClusterIP (default) exposes the service on an internal IP only reachable within the cluster.
NodePort exposes the service on each Node's IP at a static port (between 30000-32767).
LoadBalancer provisions an external load balancer in the cloud provider (AWS, GCP, etc.) and routes traffic to the NodePort automatically.

Tricky Part: NodePort actually creates a ClusterIP under the hood. LoadBalancer creates a NodePort under the hood. They build on top of each other!
6. Are Kubernetes Secrets actually secure?

By default, no. They are just base64-encoded strings, not encrypted. Anyone with access to the API or the underlying etcd database can easily decode them.

Tricky Part: Base64 is encoding, not encryption! For real security, you must explicitly configure Encryption at Rest for etcd, or use external secret management solutions like HashiCorp Vault or AWS Secrets Manager.
