GitHub Interview: Must Know & Tricky

Top version control questions focusing on edge cases, disaster recovery, and advanced commands.

Gentle Reminder: You don't have to master it all today. The secret to learning is simply focusing on deeply understanding one problem at a time until it clicks effortlessly. Small, atomic habits build unbreakable mastery.
1. How do you undo a commit that has already been pushed and made public?

You should use git revert <commit_hash>. This creates a new commit that undoes all the changes made in the specified commit, preserving the history.

Tricky Part: Never use git reset on a public/shared branch, because it rewrites history and will break the repository for your teammates when they try to pull.
2. Git Rebase vs Git Merge: When to use which?

Merge takes the contents of a source branch and integrates them with a target branch, creating a new "merge commit". It preserves the complete history and chronological order.

Rebase moves the entire feature branch to begin on the tip of the master branch, rewriting the project history to be perfectly linear.

Tricky Part: The golden rule of rebase is to never rebase a public branch. Only rebase your local, private branches before merging them into a shared branch to keep a clean history.
3. You realize you forgot to add a file to your last commit, or made a typo in the commit message. How do you fix it without creating a new commit?

Use git commit --amend. First, stage the forgotten file using git add, then run the amend command. It will open your editor to let you change the last commit message and bundle the new changes into that single commit.

Tricky Part: This rewrites the commit hash. If you've already pushed the previous commit, amending it will cause conflicts unless you force push (which you shouldn't do on a shared branch).
4. What is `git cherry-pick` and when is it useful?

git cherry-pick <commit_hash> applies the changes introduced by an existing commit from another branch onto your current branch.

Tricky Part: It's most commonly used for backporting bug fixes. If a bug is fixed in `master`, you can cherry-pick that specific fix commit to an older `release-1.0` branch without merging all the other new features.
5. You need to switch branches urgently, but have half-finished, untracked work. How do you stash it?

A standard git stash only saves changes to tracked files. To include untracked (new) files, you must use git stash -u (or --include-untracked).

Tricky Part: If you use just `git stash`, your new files will remain in your working directory and might conflict with the branch you switch to!
6. How do you find which commit introduced a bug (regression)?

Use git bisect. It uses a binary search algorithm to quickly find the offending commit.
1. git bisect start
2. git bisect bad (current broken state)
3. git bisect good <old_hash> (a known working commit)
Git will then checkout a commit in the middle for you to test.

Tricky Part: You can automate this! If you have a test script, you can run git bisect run ./test.sh and Git will find the bad commit automatically without manual intervention.
